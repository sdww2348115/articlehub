# Skill 深度分析：to-issues

> 来源：mattpocock/skills
> 分析日期：2026-04-28

---

## 一、Skill 的作用和核心功能

### 1.1 定位与目标

`to-issues` 是一个将**计划、规格文档或 PRD（产品需求文档）**拆解为独立可领取的 GitHub Issues 的 Skill。它的核心目标是解决"需求文档到手，却不知从何下手"的常见开发困境——把抽象的计划落地为具体、可执行、可追踪的任务卡片。

### 1.2 核心功能拆解

| 步骤 | 名称 | 具体动作 |
|------|------|----------|
| Step 1 | **Gather context** | 从对话上下文中获取需求内容；若用户提供了 GitHub Issue 编号或 URL，则通过 `gh issue view` 抓取完整内容和评论 |
| Step 2 | **Explore codebase**（可选） | 若尚未探索代码库，则先了解当前代码状态，确保拆分出的 issue 与现有架构匹配 |
| Step 3 | **Draft vertical slices** | 将计划拆解为"追踪弹"（tracer bullet）式的垂直切片，每个切片横穿所有集成层（schema、API、UI、测试），**而不是**按层拆分（避免 horizontal slice） |
| Step 4 | **Quiz the user** | 将拟议的拆分方案以结构化列表呈现给用户，逐项确认粒度、依赖关系、合并/拆分需求、 HITL/AFK 分类是否准确；**迭代直到用户批准** |
| Step 5 | **Create GitHub issues** | 按依赖顺序（blocker 优先）依次创建 Issue，便于在"Blocked by"字段中引用真实的 Issue 编号 |

### 1.3 关键概念：HITL vs AFK

- **HITL**（Human-In-The-Loop）：需要人工介入的切片，如架构决策、设计评审、依赖外部团队确认的事项。
- **AFK**（Away-From-Keyboard）：可独立实现并合并的切片，无需人工干预。

这一分类直接决定了任务的**可并行性**和**优先级排序**，是非常实用的二元决策模型。

### 1.4 关键概念：Vertical Slices（垂直切片）

不同于传统的按层拆分（先做完所有 API、再做所有 UI……），垂直切片强调：

> **每个切片都是一条完整的端到端路径。**

即：一个小功能点，在 schema、API、UI、测试每一层都有对应的实现，且完成后**可演示、可验证**。

## 二、写得好的地方

### 2.1 设计亮点

#### ① 明确的问题解决导向

Skill 开篇即点明解决的是"把计划变成独立可领取的 issues"这个具体问题，而不是泛泛的"辅助开发"。这种**问题即产品**的思维，让 Skill 的行为边界非常清晰——做什么、不做什么，一目了然。

#### ② 流程内嵌用户确认节点

Step 4（Quiz the user）是整个 Skill 最聪明的设计之一。它不是在最后才给用户一个"确认框"，而是在**草稿阶段就强制进行双向对话**：

- Agent 提出方案
- 用户审查粒度、依赖、分类
- 迭代直到批准

这避免了 Agent 一厢情愿拆分、用户事后不满意要返工的问题。这本质上是一个**human-in-the-loop 的迭代循环**，将用户纳入工作流的主动参与者而非被动接收者。

#### ③ HITL/AFK 二元分类的实用性

这个分类极其简洁，却解决了项目管理和团队协作中的真实痛点：
- HITL 标记了"瓶颈点"，帮助识别哪些事项在等待人工决策
- AFK 标记了"可并行推进"的事项，优化团队调度

#### ④ `<vertical-slice-rules>` 标签化规则

用 XML 风格标签包裹核心规则，是一个很巧妙的**元编程思路**：

```xml
<vertical-slice-rules>
- Each slice delivers a narrow but COMPLETE path...
- A completed slice is demoable or verifiable on its own
- Prefer many thin slices over few thick ones
</vertical-slice-rules>
```

这让规则本身可以被解析、提取、引用，甚至可以用于自动化验证。这些标签本质上是**Skill 内部的 API 契约**，定义了 Skill 的内部协议。

#### ⑤ 按依赖顺序创建 Issue

Step 5 明确要求**按 blocker 顺序创建 Issue**，这样可以在创建过程中立即获得真实的 Issue 编号，填入后续 Issue 的 "Blocked by" 字段。这避免了"用占位符 #99，后续再改"的尴尬，也保证了 Issue 之间的引用关系在创建时就已是事实。

#### ⑥ Issue 模板的结构化设计

提供的 Issue 模板包含：
- **Parent**：来源追溯（若来自 GitHub Issue 则引用）
- **What to build**：端到端行为描述（而非实现细节）
- **Acceptance criteria**：可勾选的验收条件
- **Blocked by**：依赖声明

这种结构既保证了信息的完整性，又强制要求作者（和 AI）用"可验证"而非"可实现"的语言来描述需求。

---

### 2.2 写作技巧

#### ① 简洁有力的语言风格

整篇 Skill 几乎没有冗余的废话。每句话都直接对应一个可执行的动作或规则。例如：

> "Prefer AFK over HITL where possible."

一句点明偏好，无须额外解释理由（理由已经在 HITL/AFK 的定义中隐含了）。

#### ② 强制性的条件语句

大量使用"如果…则…"的结构，例如：

> "If the user passes a GitHub issue number or URL as an argument, fetch it..."

这种写法让 Skill 的行为具有**确定性**——在什么条件下触发什么行为，不存在歧义。

#### ③ 适当的可选标注（optional）

Step 2 明确标注"（可选）"，避免了将所有步骤都设为必需的教条感。这种**对可选步骤的诚实标注**提升了 Skill 的可信度和可用性。

#### ④ 结构化呈现 Quiz 结果

Step 4 的输出格式是固定的四问：

- 粒度是否合适？
- 依赖关系是否正确？
- 是否需要合并或拆分？
- HITL/AFK 分类是否准确？

这种**固定格式的审查清单**让用户不需要自己想"还要检查什么"，降低了认知负担。

#### ⑤ 文档末尾的"红线"声明

> "Do NOT close or modify any parent issue."

这是一条明确的**禁止性规则**，以大写和强调的语气写出，让人一眼就能注意到。这种对边界的清晰声明，是好 Skill 写作的重要特征。

## 三、为我们构建 Skill 带来的提示与借鉴

### 3.1 架构层面的借鉴

#### ① 每个 Skill 应该解决一个**具体问题**

`to-issues` 的 description 就是最好的示范：

> "Break a plan, spec, or PRD into independently-grabbable GitHub issues using tracer-bullet vertical slices."

一个好的 Skill 描述应该让用户**一看就知道什么时候该用它**，而不是泛泛的"辅助编码"。

#### ② 流程 = 状态机，而非线性文档

当前很多 Skill 只是把步骤写成 Markdown 列表。而 `to-issues` 的 5 个步骤实际上构成一个**有条件分支的状态机**：

- Step 1 和 2 是准备阶段
- Step 3 是执行阶段（可重复）
- Step 4 是确认节点（循环直到批准）
- Step 5 是收尾阶段

我们在设计 Skill 时，应该明确标出**循环节点**（Step 4 迭代）、**可选节点**（Step 2 可选）和**强制节点**（Step 5 必执行）。

#### ③ 清晰的输入 → 内部处理 → 可验证的输出

`to-issues` 的输入是计划/PRD，输出是创建好的 GitHub Issues，整个过程有明确的可验证终点。我们在设计 Skill 时也应该确保：**Skill 完成后，用户知道它做了什么、结果在哪里**。

---

### 3.2 写作层面的借鉴

#### ① 用结构化标签定义内部协议

`<vertical-slice-rules>` 和 `<issue-template>` 这种 XML 风格的内嵌标签，可以作为 Skill 内部的"小语言"。这提示我们：Skill 文档不仅是给人看的，也可以设计成**机器可解析的结构**。

#### ② 用户确认节点是不可省略的安全阀

Step 4 的 Quiz 机制是一个非常值得借鉴的**设计模式**——对于任何有创意输出（拆解方案、代码生成、文档撰写）的 Skill，都应该在正式执行前设置一个**用户确认节点**。这既是对用户意图的尊重，也是对 AI 幻觉风险的主动防范。

#### ③ 禁止性规则应该醒目

"Do NOT..." 规则用大写、单独成段的方式呈现，视觉效果突出。我们在编写 Skill 时，对于**破坏性操作**（删除、覆盖、修改现有文件等）也应该用同样醒目的方式标注。

#### ④ 分类标签让 Skill 更精准

HITL/AFK 分类本质上是给任务打上了**元标签**。我们在设计 Skill 时，也可以考虑为输出的内容设计类似的元标签系统——例如代码生成的 Skill 可以标注"是否需要测试"、"是否涉及数据库迁移"等，让用户可以更精细地控制 Skill 的行为。

---

### 3.3 流程设计层面的借鉴

#### ① "先理解，再拆分"的顺序不可颠倒

Step 1 和 Step 2 实际上是**理解阶段**，Step 3 才是**执行阶段**。这个顺序不能颠倒。一个常见的 Skill 设计反模式是：直接开始执行，完全跳过理解上下文。在 `to-issues` 中，即使上下文已经存在（Step 1），代码库探索（Step 2）也是可选但被认真设计过的节点。

#### ② 迭代确认而非一次性输出

大多数 AI 输出是一锤子买卖——生成完就结束。`to-issues` 最有价值的设计是**迭代循环**：Agent 输出 → 用户反馈 → 修正 → 再次输出 → 循环直到批准。这比直接输出的质量高得多，也更符合实际工作中"方案需要讨论"的现实。

#### ③ 依赖排序决定了执行顺序

创建 Issue 时按 blocker 优先的顺序进行，这是一个**拓扑排序**思想的体现。在我们的 Skill 设计中，如果涉及多步骤创建/修改操作，也应该考虑**依赖顺序**，确保前置步骤完成后再执行后续步骤。

---

### 3.4 可改进之处（批判性思考）

一个客观的分析也应该指出不足：

1. **Step 2"探索代码库"过于简略**：没有说明如何探索、用什么命令、探索到何种程度。这可能导致 AI 行为不一致。
2. **没有错误处理机制**：如果 `gh issue view` 失败怎么办？如果 GitHub API 报错怎么办？缺少对异常情况的处理说明。
3. **Quiz 阶段的停止条件不够明确**：迭代到什么时候算"用户批准"？是否有明确的确认机制？
4. **对多成员团队场景考虑不足**：没有涉及 Issue 分配（assignee）、标签（labels）、里程碑（milestone）等团队协作相关的操作。

---

## 四、总结

`to-issues` 是一个**流程严谨、结构清晰、用户参与度高**的 Skill 典范。它的核心价值不仅在于"拆分任务"这件事本身，更在于它示范了一种**如何设计 AI 工作流**的思路：

- **以问题为中心**而非以工具为中心
- **以用户确认为安全阀**而非盲目执行
- **以可验证为终点**而非生成即结束
- **以结构化为内部协议**而非自由文本

这些设计原则，对于我们构建任何复杂度的 Skill，都具有直接的参考价值。
