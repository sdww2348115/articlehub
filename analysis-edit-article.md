# Skill 深度分析：edit-article

> 来源：[mattpocock/skills](https://github.com/mattpocock/skills) — `edit-article`
> 分析日期：2026-04-28

---

## 一、Skill 的作用和核心功能

### 1.1 定位与目标

`edit-article` 是一个**文章编辑与润色**类 Skill，面向的场景是：用户有一篇草稿文章，希望 AI 帮助其改进结构、提升可读性、压缩表达。

它的目标不是重写全文，而是**在保留原意的前提下优化表达**——这是它与 `write-article` 的本质区别。

### 1.2 核心流程拆解

| 步骤 | 动作 | 说明 |
|------|------|------|
| **Step 1** | **切分文章 → 识别 sections** | 按现有标题将文章解构为多个区块 |
| **Step 1** | **分析信息依赖关系（DAG 思维）** | 判断哪些信息必须先出现、哪些可以后出现 |
| **Step 1** | **确认分区方案（用户确认机制）** | 在动手之前先让用户审核计划 |
| **Step 2** | **逐 section 重写** | 对每个区块单独优化：清晰度、连贯性、行文流畅度 |
| **Step 2a** | **每段不超过 240 字符** | 硬性约束，强制精简 prose |

### 1.3 信息架构思维（核心亮点）

Skill 引入了一个非常精妙的隐喻：

> **"Information is a directed acyclic graph"**

这句话背后是一个深刻认知：**文章不是线性列表，信息与信息之间存在依赖图**。先决信息必须先讲，否则读者会感到困惑。这使得 AI 在编辑时不仅关注单段文字质量，还主动考虑**全局逻辑顺序**，这是大多数 AI 写作助手中缺失的能力。

## 二、写得好的地方

### 2.1 极简但完整的工作流

整个 Skill 只有 **2 个大步骤**，没有冗余的引导、示例或备选分支。相比之下，很多 Skill 写得过于冗长，塞入大量示例导致：

- AI 执行时容易迷失在细节里
- 变更需求时改动点多
- 可读性反而下降

`edit-article` 的设计哲学是：**用最少的文字覆盖最完整的决策路径**。

### 2.2 强制确认机制（Human-in-the-Loop）

```
Confirm the sections with the user.
```

这一句话实现了**人工审核节点**。在 AI 真正动手重写之前，必须先向用户确认分区方案。这意味着：

- 用户对文章结构拥有最终决定权
- 减少 AI "自作主张" 导致方向偏差的风险
- 将编辑流程从"一次性生成"变为"迭代式协作"

这比我们很多 Skill 中直接执行不确认的做法要更稳妥。

### 2.3 段落长度硬约束

```
Use maximum 240 characters per paragraph.
```

240 字符（约 120 个中文汉字）是一个经过验证的阅读友好阈值。引入这个硬约束：

- 迫使 AI 砍掉冗余表达
- 强迫每一段都保持"一个核心信息点"
- 提升文章整体的呼吸感

这个约束本身就是一个**编辑原则的具体化**，而非模糊的"写简洁一点"。

### 2.4 清晰的步骤层级编号

```
1. First, ...
2. For each section:
   2a. Rewrite the section...
```

使用 `1.` / `2a.` 这种层级编号，与 NLPChain 中的 "Thought/Action/Action Input/PAUSE/PAUSE" 类似——明确告诉 AI 执行顺序和嵌套关系。AI 在执行时不会混淆步骤的前后依赖。

### 2.5 精准的触发词设计

Description 部分：

> "Edit and improve articles by restructuring sections, improving clarity, and tightening prose. **Use when user wants to edit, revise, or improve an article draft.**"

最后一句**明确指出触发时机**——"当用户想要编辑、修改、完善文章草稿时"。这种触发词设计让 Skill 的调用边界非常清晰，避免被误用于初次写作场景。

## 三、为我们构建 Skill 能带来的提示与借鉴

### 3.1 工作流设计层面

#### ✅ 启示 1：把"分析-确认-执行"三段式作为复杂任务的标准模式

```
分析阶段（AI 自主完成结构规划）
  ↓
确认阶段（Human-in-the-Loop）
  ↓
执行阶段（按确认的计划逐项完成）
```

对于涉及内容创作的 Skill（如文章、文档、PPT 等），这个三段式能显著降低 AI 输出的方向性风险。`edit-article` 是这个模式的极佳范例。

#### ✅ 启示 2：在 Skill 中嵌入"硬性质量约束"

不要只写"提升清晰度"这种模糊要求，而是给出**可量化的约束**（如 240 字符上限）。这样 AI 执行时更容易判断输出是否达标，也方便我们后续评估 Skill 效果。

#### ✅ 启示 3：使用 DAG 思维处理有依赖关系的任务

当一个复杂任务可以被分解为多个子步骤，且子步骤之间存在先后依赖时，**显式声明这个依赖关系**（像 `edit-article` 提到 DAG 那样）能帮助 AI 更合理地安排执行顺序。我们在构建类似 Skill 时，可以主动识别这类依赖结构。

### 3.2 写作与格式层面

#### ✅ 启示 4：Skill 文档本身也是 prose——要简洁、有结构、供 AI 阅读

`edit-article` 的文字量极小（不到 100 词），但该有的信息一个不缺。这提醒我们：

- **Skill 文档不是设计文档，不需要铺陈背景**
- **直接写"做什么"和"怎么做"，砍掉所有解释性废话**
- AI 读 Skill 文档的时间成本直接影响执行效率

#### ✅ 启示 5：善用"换行+缩进"表达层级

原文用缩进表达 `2a` 是 `2` 的子步骤，这种格式在纯文本 Skill 中非常有效。我们在写 Skill 时也应该：

```
主步骤（无缩进）
  子步骤（1-2 空格缩进）
    具体指令（2 空格缩进）
```

### 3.3 架构与工程层面

#### ✅ 启示 6："编辑类" Skill 的设计模式可以复用

`edit-article` 的核心模式是：**解构 → 分析 → 确认 → 逐项重写**。这个模式可以迁移到很多场景：

| 场景 | Skill 名称（建议） | 迁移要点 |
|------|------------------|----------|
| 聊天记录总结 | `summarize-chat` | 按人/时间切分 → 确认 → 逐段摘要 |
| 文档翻译 | `translate-doc` | 按章节切分 → 确认 → 逐节翻译 |
| 代码重构 | `refactor-code` | 按函数/模块切分 → 确认 → 逐项重构 |
| 演讲稿优化 | `improve-speech` | 按段落切分 → 确认 → 逐段润色 |

#### ✅ 启示 7：触发词描述要包含"用户语言"

Description 中"edit, revise, improve an article draft"这些词都是普通用户在日常表达中会说的。不是技术术语，而是**用户需求语言**。Skill 的 description 应该用用户的词汇来描述触发条件，而非用 AI自己能理解的抽象能力描述。

### 3.4 值得注意的局限性

我们在借鉴时也需要注意 `edit-article` 本身的不足：

- **没有处理多媒体内容**：图片、表格、代码块等如何编辑没有涉及
- **没有处理多语言场景**：假设原文和目标读者使用同一语言
- **段落长度约束可能过于严格**：某些类型的文章（如学术论文）需要更长的论证段落
- **没有"反向编辑"模式**：即从编辑状态还原到草稿状态（版本回溯）

这些局限性可以作为我们构建 Skill 时的**扩展点**来考虑。

---

## 四、总结

`edit-article` 是一个**麻雀虽小、五脏俱全**的 Skill 范例。它的价值不仅在于功能本身，更在于它展示了一种**简洁而严谨的 Skill 设计哲学**：

1. **少即是多** — 用最少的文字覆盖完整的执行路径
2. **约束即质量** — 硬性约束比模糊要求更容易执行和评估
3. **人工审核节点** — 复杂内容任务必须保留用户确认环节
4. **结构化思维** — 将任务分解为 DAG 式的依赖图
5. **用户语言** — 触发词描述使用用户自然表达

这些原则值得在构建任何 Skill 时反复参照。
