## 一、Skill 的作用和核心功能

### 定位

`scaffold-exercises` 是一个**结构化目录脚手架工具**，专门用于快速创建课程练习的目录结构。它不是一个执行某种算法或调用外部 API 的工具，而是一个**纯文件/目录操作**的指南性 Skill。

### 核心功能

将用户（如课程维护者）的自然语言指令（"我想创建一节关于 BM25 的练习"）转化为符合规范的目录树：

```
exercises/
└── 01-retrieval-skill-building/
    └── 01.03-retrieval-with-bm25/
        ├── problem/
        │   ├── readme.md
        │   └── main.ts
        ├── solution/
        │   ├── readme.md
        │   └── main.ts
        └── explainer/
            └── readme.md
```

### 关键约束

文档中内嵌了 lint 规则——这是一个**自验证机制**。Skill 不仅告诉用户"怎么做"，还告诉用户"怎么验证做的对不对"。lint 规则包括：
- 必须存在哪些子文件夹
- readme.md 不能为空、不能有死链
- 不能有 `.gitkeep` / `speaker-notes.md`
- 不能在 readme 里写 `pnpm run exercise`
- 有代码的子文件夹必须有 `main.ts`

## 二、写得好的地方

### 1. 极致的上下文压缩——用最小篇幅传递最精准的约束

整个 Skill 只有约 120 行（包含代码块），却覆盖了：
- 命名规范（编号规则 + dash-case）
- 变体类型（problem/solution/explainer 三大类）
- 文件规范（readme.md 的内容约束）
- 工作流（5 步）
- Lint 规则（8 条，含边界情况）
- 移动/重命名专项指南

这种信息密度背后是**精确性思维**：每一个规则都是曾经踩过的坑，每一句话都在消除歧义。

### 2. 命令锚定（Command Anchoring）——内置验证，减少幻觉

> `pnpm ai-hero-cli internal lint`

这句话是 Skill 的"执行锚点"。AI 在完成目录创建后运行这条命令，如果报错就迭代修复。这个设计：
- 让 AI 有明确的结束判断（lint 通过 = 完成）
- 减少了"我以为做对了"的幻觉
- 把 lint 规则从"文字描述"变成了"可执行验证"

**写作技巧**：在描述复杂规范时，给出一个可运行的验证命令，比长篇文字描述更有效。

### 3. 命名层次清晰——零歧义的编号系统

```
Sections: 01-retrieval-skill-building    (两位数 + dash-case)
Exercises: 01.03-retrieval-with-bm25     (四位前缀.两位后缀 + dash-case)
```

文档用**数学表达式的类比**来解释编号逻辑：
> Section number = `XX`, exercise number = `XX.YY`

这种写法比"节是两位数，节内练习是点分两位数"更无歧义。

### 4. 默认值明确——消除"该选哪个"的决策疲劳

> When stubbing, default to `explainer/` unless the plan specifies otherwise.

在描述三个变体（problem/solution/explainer）后，给出了一个**默认推荐**。这对 AI 行为一致性至关重要——当用户说"创建一个练习"而不指定类型时，AI 不需要凭空猜测。

### 5. 边界情况处理——专门章节

文档末尾有一个**专门针对"移动/重命名"的专项指南**，这说明：
- 作者预见到了目录创建后的二次操作场景
- 重命名涉及 `git mv` 的细节，是真实工作流的一部分

### 6. 文档结构：倒三角布局

```
开篇：目的 + 命令锚点
↓ 中层：操作规范（命名、文件类型）
↓ 底层：细节规则（lint summary）
↓ 专项：移动/重命名
```

越重要的信息越靠前。这是一种**优先级驱动的文档结构**，而不是逻辑驱动的（后者会把"lint 规则"放在最前面，吓跑读者）。

## 三、为我们构建 Skill 能带来的借鉴

### 借鉴 1：自验证 > 描述性规则

我们很多 Skill 的规范是"请确保字段非空"这类文字描述。但更好的方式是：**给一个可执行的验证命令**，让 AI 自己确认是否合规。

例如，做飞书任务创建 Skill 时，可以指定"创建后调用 get 接口验证返回值中 task_guid 存在"作为验证步骤。

**行动项**：在编写 Skill 时，检查是否有可嵌入的验证命令（或 API 调用），用验证替代描述。

### 借鉴 2：默认值即决策——消除不确定性

`scaffold-exercises` 明确说了"stubbing 时默认 explainer"，这让我们思考：每个多选项 Skill 是否都应该有**明确默认值**？

对于飞书日历 Skill：
- 创建日程时，若用户没说持续时间，默认 1 小时还是 30 分钟？
- 若没指定时区，默认用户本地还是 UTC？

**行动项**：检查我们现有的 Skill，对于有多种可选方案的步骤，加上"若未指定，默认 X"的说明。

### 借鉴 3：命名空间设计是脚手架 Skill 的核心

这个 Skill 花了大篇幅讲编号规则（`01`、`01.03`），因为**一旦编号错了，整个目录结构就废了**。这启示我们：

- 如果我们要做一个"创建飞书多维表格"的 Skill，表格/字段的命名规范同样重要
- 编号体系要能用正则表达，这样 AI 才能验证用户输入是否符合规范

### 借鉴 4：Lint 内嵌是小型 Skill 的高阶玩法

`pnpm ai-hero-cli internal lint` 是一个内部工具的 lint 命令。文档把它当作**验证机制**来用，这是 Skill 设计的一个高阶模式：

```
1. 描述要做什么（规范化描述）
2. 给一个验证命令（验证性描述）
3. 让 AI 迭代直到通过（自主纠正）
```

我们不一定需要 `pnpm` 或内部 CLI，但可以用类似的思路：
- 调用实际 API 检查结果
- 用正则/脚本验证输出结构

### 借鉴 5：状态机思维——覆盖完整生命周期

`scaffold-exercises` 不仅讲了"创建"，还讲了"移动/重命名"。这意味着它在用**状态机思维**：目录经历了 创建 → 使用 → 移动/重命名 → （可能的）删除 等状态。

我们在设计 Skill 时，也可以思考：
- 用户在执行这个操作前需要什么状态？
- 操作后会进入什么状态？
- 有哪些二次操作路径？

### 借鉴 6：边界示例比通用描述更有价值

文档给出了：
- ✅ `01-retrieval-skill-building`（合法命名）
- ✅ `01.03-retrieval-with-bm25`（合法命名）
- ❌ `.gitkeep` 文件禁止
- ❌ `speaker-notes.md` 文件禁止

**正反示例对比**比"请使用 dash-case 格式"更不容易出错。我们在编写 Skill 时，可以多收集"错误案例"，让 AI 学会识别禁区。

## 四、总结

`scaffold-exercises` 是一个**信息密度极高、操作精确、自我验证**的 Skill。它不依赖复杂的 API 调用，只依赖文件系统操作和一条 lint 命令，却设计出了可靠的工作流。

它的核心设计哲学可以概括为：

| 维度 | 设计选择 |
|------|---------|
| 验证方式 | 可执行命令 > 文字描述 |
| 默认值 | 明确默认值 > 让 AI 自己猜 |
| 生命周期 | 覆盖创建 + 二次操作 |
| 信息结构 | 优先级驱动（目的 → 规范 → 细节） |
| 错误预防 | 正反示例 > 模糊描述 |

这些原则适用于我们所有 Skill 的设计，无论是飞书日历、任务管理还是文档创建。
