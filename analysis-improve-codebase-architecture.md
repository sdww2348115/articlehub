# Skill 分析：improve-codebase-architecture

> 来源：[mattpocock/skills](https://github.com/mattpocock/skills) — `improve-codebase-architecture`

---

## 一、Skill 的作用和核心功能

### 1.1 定位

这是一个**代码架构改进辅助 Skill**。它不是直接帮你写代码，而是作为 AI Agent 的"架构审查搭档"，引导用户发现代码库中值得深化的模块，并一步步推进重构决策。

核心目标：
- **可测试性（Testability）**——让模块的接口成为测试表面
- **AI 可导航性（AI-navigability）**——让 AI 能顺畅理解和操作代码库结构

### 1.2 核心功能

#### (1) 深度发现（Deepening Opportunities）

该 Skill 发明了一个精准的概念——**深度（Depth）** vs **浅度（Shallow）**：

- **深模块（Deep Module）**：接口小而克制，实现丰富。调用者获得高杠杆（leverage），维护者获得高局部性（locality）。
- **浅模块（Shallow Module）**：接口几乎和实现一样复杂。没有存在的意义，是典型的"过渡设计"产物。

Skill 的工作就是发现"浅模块"，提出"深化"方案。

#### (2) 统一语言体系（Glossary + LANGUAGE.md）

Skill 定义了一套精确的术语表，强制在所有建议中使用：

| 术语 | 含义 |
|------|------|
| Module | 任何有接口和实现的东西（函数、类、包、slice） |
| Interface | 调用者必须知道的一切：类型、不变量、错误模式、顺序、配置 |
| Depth | 接口上的杠杆：大量行为浓缩在小接口背后 |
| Seam | 接口所在的位置——可以在不原地编辑的情况下改变行为 |
| Adapter | 在某个 seam 上满足接口的具体事物 |
| Leverage | 调用者从深度中获得的东西 |
| Locality | 维护者从深度中获得的东西：变更、bug、知识集中在一处 |

**禁止使用**：component、service、API、boundary 等模糊词汇。

#### (3) 基于项目语境的探索

Skill 读取项目自带的 `CONTEXT.md`（领域语言）和 `docs/adr/`（架构决策记录），确保：
- 建议使用项目自己的术语（"Order intake module"而非"FooBarHandler"）
- 不重新争论已记录的架构决策（ADR）

#### (4) 三阶段流程

```
Explore → Present Candidates → Grilling Loop
```

- **Explore**：先读文档，再用 Explore 子代理遍历代码库，寻找浅模块、紧耦合、缺乏局部性等摩擦点
- **Present Candidates**：列出编号的深化机会点（不提前给接口方案）
- **Grilling Loop**：用户选一个后，深入讨论设计树，同步更新 CONTEXT.md 或记录 ADR

## 二、写得好的地方

### 2.1 设计亮点

#### (1) "Deletion Test"——简洁而深刻的判断法则

> 想象删除这个模块。如果复杂性随之消失，它是 passthrough。如果复杂性分散到 N 个调用者身上，它在自证其价值。

这比大多数"高内聚低耦合"的抽象教条更具体、更可操作。一个简单的心智测试，立刻区分"有意义模块"和"过渡抽象"。

#### (2) "One adapter = hypothetical seam. Two adapters = real seam."

接口设计何时才算"真实"？当且仅当有两个以上的具体实现时。这是一个**反教条化**的洞见——不是在设计阶段假设接口应该存在，而是用实际需求（第二个适配器）来验证接口的必要性。

#### (3) Side Effects 作为对话的自然延伸

Skill 并没有把"更新文档"当作额外步骤，而是作为 Grilling Loop 中的自然副作用：
- 讨论中命名了新概念 → 顺手写入 CONTEXT.md
- 用户给出了有分量的拒绝理由 → 顺势提议写 ADR

这体现了**文档即思考**的理念：架构讨论和文档演化同步发生。

#### (4) ADR 冲突只在该摩擦真实存在时才提出

Skill 明确说：不要列出每个被 ADR 禁止的理论重构。只有当摩擦足够真实、值得重开 ADR 讨论时，才标注矛盾。

这是对"文档权威性"的尊重，也是避免无谓讨论的工程判断。

#### (5) "Do NOT propose interfaces yet"——克制住直接给答案的冲动

在候选阶段只描述问题和解决方向，不提前进入接口设计。这给了用户选择权和思考空间，也符合真实的架构工作流——先确认"要不要改"，再讨论"怎么改"。

#### (6) 与项目语境深度绑定

Skill 不是通用建议，而是读取 `CONTEXT.md` 和 `docs/adr/`，使用项目自己的语言。这避免了 AI 给出的建议和项目已有认知产生割裂感。

---

### 2.2 写作技巧

#### (1) 术语精炼，定义精准

每个术语都有**操作性定义**，而非描述性定义。例如 Depth 不是"做得多"，而是"interface 上的 leverage"。这种定义方式让术语可直接用于判断。

#### (2) 流程指令清晰，动作明确

每个阶段都有明确的：
- **输入**（读什么、用什么工具）
- **禁止项**（不做什么，如"不主动提示创建文档"）
- **输出格式**（用什么结构描述候选点）

#### (3) 克制而自信的语气

Skill 使用大量**命令式语句**（"Present a numbered list"、"Do NOT propose interfaces yet"），但没有居高临下的说教感。语气克制，规则清晰，执行边界明确。

#### (4) 元认知暴露

Skill 主动说明"为什么要这样做"，而非仅仅"做什么"：
- "Consistent language is the point"
- "The interface is the test surface"
- 解释了为什么禁止某些词汇（避免漂移）

#### (5) 结构化输出示例（内嵌在描述中）

候选点的描述格式（Files / Problem / Solution / Benefits）直接内嵌在描述文本中，说明了输出应该是什么结构。这种"格式预埋"让执行者无需额外猜测。

## 三、为我们构建 Skill 能带来的提示/借鉴

### 3.1 技能设计层面

#### (1) 先定义语言，再定义流程

好的 Skill 应该先建立一套一致的术语体系，再在其上构建流程。这比直接给流程清单更有生命力——因为所有参与者（人类和 AI）都共享同一套词汇。

**借鉴**：在我们设计新 Skill 时，先花时间定义 Glossary，并要求整个 Skill 严格使用这些术语。

#### (2) 引入可操作的"判断法则"

"Deletion Test"和"Two Adapters Rule"都是可操作的判断法则，而非泛泛的原则（"要解耦"、"要测试"）。

**借鉴**：每个 Skill 应至少包含一个这样的**快速判断准则**，让 AI 在执行时能快速做决策，而不必每次都重新推理。

#### (3) 三阶段流程（探索→候选→深化）是可复用的模式

```
先广泛探索 → 再提炼候选 → 再深入单个目标
```

这个模式适用于任何需要"从粗到细"渐进式工作的 Skill。Explore 阶段可以快速并行，Present 阶段给用户决策点，Grilling 阶段允许深度挖掘。

**借鉴**：对于需要多轮交互的 Skill，考虑这种三阶段结构。

#### (4) Side effects 的设计哲学——文档是副产品，不是目标

Skill 没有单独设置"更新文档"的步骤，而是把文档更新作为决策讨论的自然延伸。

**借鉴**：我们在构建 Skill 时，问自己"这个 Skill 的自然副产品是什么？"并允许这些副产品在适当的对话节点自然产生。

#### (5) Skill 应知道自己与项目其他文档的关系

Skill 明确引用了 `LANGUAGE.md`、`CONTEXT-FORMAT.md`、`ADR-FORMAT.md` 等文件，承认自己是更大系统的一部分。

**借鉴**：如果一个 Skill 依赖其他 Skill 或文档，应在 Skill 中明确声明引用关系，而不是假设用户知道如何整合。

---

### 3.2 写作规范层面

#### (1) 禁止项和例外同等重要

Skill 明确列出了**不要做**的事（"don't flag their absence"、"Don't list every theoretical refactor an ADR forbids"）。

**借鉴**：我们的 Skill 文档应同等重视"禁止项"，因为这些往往是 AI 最容易漂移的地方。

#### (2) 区分"ephemeral reasons"和"load-bearing reasons"

Skill 在 ADR 创建条件中做了这个区分——只有"load-bearing"理由才值得记录为 ADR。

**借鉴**：在涉及"何时创建文档/记录"的逻辑时，明确区分暂时代码和持久记录。

#### (3) 描述输出格式，而不是展示示例

Skill 用"Files / Problem / Solution / Benefits"说明每个候选点的结构，而不是给一个 Markdown 示例。

**借鉴**：描述格式用结构化字段优于展示完整示例——更精确，也更通用。

#### (4) "Proceed silently"的智慧

如果 `CONTEXT.md` 不存在，Skill 不主动提示创建，只是静默继续。这是对"不要强制工作流"原则的体现。

**借鉴**：在 Skill 中加入"当 X 不存在时，静默处理"的条件，避了一味要求用户先做准备工作。

---

### 3.3 整体架构层面

#### (1) Skill 是对话协议，不是执行脚本

这个 Skill 没有试图覆盖所有情况，而是设计了"用户选哪个 → 我们深入聊"的交互模式。Skill 本质上是一个**对话协议**，定义了不同阶段的参与规则。

**借鉴**：我们的 Skill 应该更多定义"对话规则"而非"完整执行路径"，留出用户主导的空间。

#### (2) 对 AI 同行（和其他 AI Agent）的尊重

Skill 引用 `INTERFACE-DESIGN.md` 说"想探索替代接口？去看那个 Skill"，这是一种 Skill 间优雅的解耦。

**借鉴**：设计 Skill 时，明确标注"其他 Skill 负责这部分"，而非试图在一个 Skill 里覆盖所有相关能力。

#### (3) 强调"深化"而非"重构"

Skill 使用"deepening opportunities"而非"refactoring opportunities"。这个措辞强调的是**把浅的做深**，而不是"推翻重来"。这降低了用户的心理阻力，也更符合真实的架构改进节奏。

**借鉴**：在 Skill 的命名和描述中，选择更正向、更精确的词汇框架。

---

## 四、总结

`improve-codebase-architecture` 是一个**高度工程化**的 Skill。它的优秀不在于覆盖了多少功能，而在于：

1. **精确的语言体系**——所有参与者共享同一套词汇
2. **可操作的判断法则**——Deletion Test、Two Adapters Rule
3. **克制的节奏设计**——不提前给答案，给用户决策空间
4. **对话即文档**——文档更新作为讨论的自然副产品
5. **对项目语境的尊重**——不重新争论已有决策

对我们构建 Skill 最有价值的借鉴是：**先建立语言，再建立流程；先定义判断法则，再执行动作；先克制输出，再渐进深化。**
