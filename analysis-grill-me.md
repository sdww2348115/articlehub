## 一、Skill 的作用和核心功能

### 1.1 定位一句话

**"强制追问式审查技能"** —— 通过不断追问，逼迫用户把计划或设计中的每一个模糊角落都暴露出来，直到双方达成真正的共识。

### 1.2 解决的问题

在实际的 AI 协作中，用户经常给出一个**笼统的计划或设计**，但其中存在大量：
- **隐含假设**（没有被显式确认的前提）
- **未解决的依赖**（A 决策依赖 B 决策，但 B 还没讨论）
- **未遍历的分支**（设计树上的某些分支从未被探索）
- **模糊边界**（谁负责什么？什么情况算"完成"？）

这些模糊地带如果不被追问，项目执行时就会不断返工、重新对齐。`grill-me` 正是为解决这一痛点而生。

### 1.3 核心功能拆解

| 功能点 | 说明 |
|--------|------|
| **系统性追问** | 不停留在表面，逐层深入到设计树的每个分支 |
| **提供推荐答案** | 每问一个问题，同时给出 AI 认为合理的答案，供用户参考或反驳 |
| **依赖顺序解决** | 决策之间有依赖关系时，先解决前置决策，再解决依赖它的后续决策 |
| **代码库探索** | 如果问题可以直接通过阅读代码来回答，就直接去读，而不是空谈 |
| **单问题模式** | 每次只问一个问题，避免信息过载，让用户逐个击破 |

## 二、写得好的地方

### 2.1 设计亮点

#### 亮点 1：**"强制追问"而不是"被动回答"**

普通的 AI 助手是"用户说什么，我回答什么"。这个 skill 的核心转变是：**AI 主动成为追问方**，承担起"质疑者"的角色。

这种角色转换非常有价值。现实中人们很难自我质疑自己的计划，总会下意识地跳过那些"感觉没问题"的地方。AI 作为外部力量，可以毫无顾忌地追问。

#### 亮点 2：**"设计树"（Design Tree）的思维框架**

> "Walk down each branch of the design tree, resolving dependencies between decisions one-by-one."

这个比喻极其精准。设计确实是一棵树：

```
                    [核心决策：架构选型]
                    /          |          \
              技术栈         部署          数据
              /    \        方案         方案
          前端      后端   ...          ...
```

每个分支都可能是一个独立的决策域，而某些分支之间存在依赖关系（先确定后端语言，才能决定 ORM 选型）。这个 skill 明确要求**按依赖顺序**解决，而不是乱序或并行追问。

#### 亮点 3：**"推荐答案"机制——降低回答成本**

> "For each question, provide your recommended answer."

这个设计非常巧妙。用户在被追问时，如果要**从零思考**每一个问题，成本很高。但 skill 要求 AI **同时给出推荐答案**，用户只需要判断"同意 or 不同意"，这大大降低了用户的回答成本，也加快了对话节奏。

#### 亮点 4：**代码库探索作为追问的替代方案**

> "If a question can be answered by exploring the codebase, explore the codebase instead."

这是对 AI 能力的充分利用。如果一个问题可以直接从代码中找到答案（比如"这个模块目前用了什么库？"），就不应该空对空地讨论，而是直接读代码。这让 AI 的追问不是纯粹的理论推演，而是**事实驱动的**。

#### 亮点 5：**触发条件写得具体、明确**

```yaml
Use when user wants to stress-test a plan, get grilled on their design, or mentions "grill me".
```

触发条件包含了三种等价表达：
1. "我想压力测试一个计划"（stress-test a plan）
2. "让我的设计被质疑"（get grilled on their design）
3. 直接说"grill me"

这样用户在各种口语化表达下都能激活这个 skill，降低了记忆负担。

### 2.2 写作技巧

#### 技巧 1：**极简 prompt，零废话**

整个 instruction 只有约 70 词（英文），分成 4 句话，每句话都承载了不可删除的信息：

1. 做什么 → 不断追问直到共识
2. 怎么做 → 走设计树的每个分支，按依赖顺序
3. 提问策略 → 每问附推荐答案
4. 节奏控制 → 一次只问一个
5. 务实例外 → 能读代码就直接读

没有一句是" filler "，没有解释为什么要这样做，就是干净的指令。

#### 技巧 2：**使用强动词驱动行为**

- `Interview`（访谈）
- `Walk down`（沿着...走）
- `Resolving`（解决）
- `provide`（提供）
- `Ask...one at a time`（一次问一个）

每一个动词都对应一个具体的 AI 行为，没有抽象的描述。

#### 技巧 3：**隐含了角色设定**

虽然没有直接说"You are a..."，但整个 prompt 让 AI 自然地成为了一个**严格的架构评审者**或**资深 CTO** 的角色。这种角色设定是通过行为描述来隐式传递的，比直接说"You are a strict reviewer"更有效，因为行为 > 身份标签。

## 三、为我们构建 Skill 能带来的提示/借鉴

### 3.1 核心借鉴：Skill 的本质是"行为模式"，不是"功能列表"

`grill-me` 之所以有效，不是因为它功能多，而是因为它定义了一个**清晰的交互模式**：

```
用户提出计划 → AI 追问 → 用户回答 → AI 再追问 → ... → 共识达成
```

我们在设计 skill 时，应该先想清楚：**这个 skill 定义的 AI 和用户之间的交互模式是什么？** 而不是罗列一堆功能点。

### 3.2 "推荐答案"机制值得借鉴

这个机制可以在很多需要用户做决策的 skill 中复用：

- **评审 skill**：AI 给出修改建议，用户确认
- **规划 skill**：AI 给出时间线，用户调整
- **写作 skill**：AI 给出初稿，用户审批

本质上是**AI 提供锚点，用户在此基础上决策**，大幅降低用户的认知负担。

### 3.3 设计树/决策树的思维框架可以迁移

任何涉及**多层级决策**的场景，都可以用"设计树"的框架来组织：

- 架构设计
- 产品功能规划
- 技术债务清理优先级
- 代码重构路线图

在 instruction 中明确要求"按依赖顺序"遍历决策树，可以让 AI 的追问更有逻辑性，而不是东一榔头西一棒槌。

### 3.4 "能读代码就读代码"是对 AI 能力的聪明使用

这一条提醒我们：**不要让 AI 只靠猜测来提问或回答**。当 AI 能够主动获取事实时，它的追问和给出的推荐答案质量会大幅提升。我们在设计其他 skill 时，也应该考虑：**有哪些问题 AI 其实可以直接获取信息，而不需要用户描述？**

### 3.5 触发条件的写法

```yaml
Use when user wants to stress-test a plan, get grilled on their design, or mentions "grill me".
```

这是一种**同义复现**的触发条件写法——同一个意图，用三种不同的表达方式列出。这比我们常见的单一触发词更鲁棒，因为用户的表达是多样化的。

**借鉴：对于核心 skill，触发条件应该列出同义的多种表达**，而不是只写一个触发词。

### 3.6 极简原则

整个 skill 只有 `description` 和 instruction 两部分，加起来不过 80 词左右。这提醒我们：

> **Skill 不是越大越全越好。越小越精准的 skill，越容易被正确调用，也越容易维护。**

我们自己在写 skill 时，应该抵制"把所有相关功能都塞进去"的冲动，聚焦在单一行为模式上。
