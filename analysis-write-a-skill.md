## 一、Skill 的作用和核心功能

### 1.1 定位

`write-a-skill` 是一个**元技能（meta-skill）**——它的作用不是完成某个具体业务任务，而是教 AI **如何构建新的 skill**。它本质上是一套结构化的 skill 创作工作流，覆盖从需求收集到交付审查的完整流程。

### 1.2 核心功能拆解

#### 功能 1：需求收集（Gather requirements）

通过向用户提问，明确四个维度：

| 维度 | 问题 | 目的 |
|------|------|------|
| 领域 | 这个 skill 处理什么任务/场景？ | 定位边界 |
| 用例 | 具体要覆盖哪些使用情况？ | 定义范围 |
| 形态 | 纯指令，还是需要可执行脚本？ | 确定技术方案 |
| 材料 | 有没有现成的参考资料要纳入？ | 复用资产 |

#### 功能 2：起草技能（Draft the skill）

产出三个层次的产物：

- **SKILL.md**（必需）：主指令文件
- **REFERENCE.md**（可选）：超过 500 行时的拆分文档
- **scripts/**（可选）：确定性操作的工具脚本

#### 功能 3：交付审查（Review with user）

用三个确认性问题做用户验证：
- 是否覆盖了你的用例？
- 有没有遗漏或不清晰的地方？
- 各部分详略是否合适？

## 二、写得好在哪里（设计亮点与写作技巧）

### 2.1 整体架构亮点

#### 亮点 1：「三步工作流」设计——极简但完整

```
收集需求 → 起草 → 用户审查
```

没有复杂的分支，没有冗余的步骤。每一步都有明确的输入和输出，形成一个**最小可用闭环**。这正是好的 agentic workflow 应该有的样子。

#### 亮点 2：「渐进式披露」（Progressive Disclosure）的结构设计

文档结构清晰地体现了渐进式披露原则：

```
SKILL.md（核心指令，精简）
    ↓（需要时）
REFERENCE.md（详细文档，补充）
    ↓（需要时）
scripts/（工具脚本，延伸）
```

主文件只承载最常用的内容，罕见的高级内容按需展开。这对 token 消耗和加载速度有直接帮助。

#### 亮点 3：「审查清单」内嵌于工作流

Review Checklist 不是放在文档末尾当装饰，而是作为工作流第 3 步的检查工具。这种**把 QA 内嵌到流程中**的做法，比事后检查更有效。

---

### 2.2 写作技巧亮点

#### 技巧 1：「反例教学法」强化描述规范

```markdown
**Good example**:
Extract text and tables from PDF files...

**Bad example**:
Helps with documents.
```

一正一反，对比鲜明，让规则无需解释即可理解。这种「对错对比」的写法比纯陈述规则要生动得多。

#### 技巧 2：「Description 是唯一入口」的认知锚定

文档花了大量篇幅讲 `description` 的重要性（独立章节、最大字符数、第三人称、触发词格式），因为作者清楚：**agent 选择 skill 的判断依据就是 description**。把这一事实明确告诉使用者，让他们从一开始就写对。

#### 技巧 3：模板即文档

提供现成的 SKILL.md 模板本身就是一种教学——使用者不需要自己推断格式，直接照着填就行。这种「COPY-PASTE」就能上手的设计，降低了技能创建的门槛。

#### 技巧 4：字数约束传递清晰价值观

- SKILL.md ≤ 100 行
- description ≤ 1024 字符

约束本身传递了一种设计哲学：**简洁 > 完备**。让使用者明白：好的 skill 不是把所有知识都塞进去，而是让核心逻辑一目了然。

---

### 2.3 文件结构设计的精妙

```
skill-name/
├── SKILL.md           # 主文件，必需
├── REFERENCE.md       # 参考文档，按需
├── EXAMPLES.md        # 示例，按需
└── scripts/
    └── helper.js      # 工具脚本，按需
```

值得注意：
- **文件名全大写**（REFERENCE.md、EXAMPLES.md）传递了一种「这是资产文件」的信号，与主文件 SKILL.md 区分
- **scripts/ 目录**的存在本身就是一个设计宣言：skill 不只是纯文本指令，也可以携带可执行能力
- **没有规定统一的子目录名**，给了使用者灵活空间

---

### 2.4 决策逻辑清晰（When to ...）

两个决策节点写得极为精准：

**When to Add Scripts** — 满足以下任一条件就加脚本：
1. 操作是确定性的（validation、formatting）
2. 同一段代码会被反复生成
3. 错误需要显式处理

**When to Split Files** — 满足以下任一条件就拆分：
1. SKILL.md 超过 100 行
2. 内容涉及不同领域
3. 高级功能只是偶尔需要

这两套规则都是**可操作、可验证的判断条件**，不是模糊的原则。这种写法让使用者拿到的是决策树，而不是哲学讨论。

## 三、为我们构建 skill 带来的启示与借鉴

### 3.1 结构层面：标准化我们的 skill 体系

目前我们的 skill 库缺乏统一的结构规范。建议采用以下层级：

```
skill-name/
├── SKILL.md           # 主指令文件（必需，≤100 行）
├── REFERENCE.md       # 深度参考资料（可选）
├── EXAMPLES.md        # 使用示例（可选）
└── scripts/           # 可执行工具（可选）
    └── *.js / *.sh
```

**建议规则**：
- 所有 skill 必须有 `SKILL.md`
- 超过 80 行的 skill 必须考虑拆分
- 涉及外部 API 调用的 skill，应在 scripts/ 中封装工具函数

### 3.2 描述（description）规范

description 是 agent 的「选择依据」，必须遵循以下规范：

| 规范项 | 要求 |
|--------|------|
| 字数上限 | 1024 字符 |
| 人称 | 第三人称 |
| 句式 | 第一句：做什么；第二句：触发条件（Use when...） |
| 触发词 | 必须包含具体关键词（文件类型、操作类型、场景） |

**反例识别**：
- ❌ "帮助用户处理文档"
- ❌ "这是一个有用的技能"
- ✅ "从 PDF 文件中提取文本和表格，填写表单，合并文档。当用户提到 PDF、表单、文档提取时使用。"

### 3.3 工作流设计：每个 skill 都应是一个闭环流程

不要只写「这个 skill 能做什么」，要明确**使用这个 skill 的步骤是什么**。参考三步法：

```
Step 1: 收集信息（向用户提问，确认范围）
Step 2: 执行任务（按照预设流程处理）
Step 3: 交付确认（展示结果，验证是否满足需求）
```

这对于复杂任务类 skill（如飞书日历、任务管理）尤其重要。

### 3.4 审查清单（Review Checklist）机制

每个 skill 交付前，应通过以下检查：

- [ ] description 包含触发词（"Use when..."）
- [ ] SKILL.md 在 100 行以内
- [ ] 无时间敏感信息（无「当前」「现在」等词）
- [ ] 术语一致
- [ ] 有具体示例
- [ ] 引用层级不超过 1 层（即 SKILL.md → REFERENCE.md，不要跨更多文件）

这条可以直接照搬，作为我们 skill 的交付标准。

### 3.5 工具脚本的嵌入策略

对于**确定性操作**（如飞书 API 调用格式化、日程时间转换），应封装为 scripts/ 中的工具函数，而非让 agent 每次重新生成。

判断标准：
- 同一操作出现 ≥ 2 次 → 封装
- 错误处理逻辑复杂 → 封装
- 涉及外部依赖（API、数据库）→ 封装

### 3.6 渐进式披露的实现模式

文档内容按使用频率分层：

| 层级 | 内容 | 文件 |
|------|------|------|
| 核心路径 | Quick Start + 常用 Workflows | SKILL.md |
| 扩展路径 | 详细配置 + 边缘场景 | REFERENCE.md |
| 参考路径 | 完整 API / Schema | EXAMPLES.md 或独立文件 |

SKILL.md 的设计哲学是：**让人第一眼就明白怎么用**，不需要读完所有文件才能开始。

## 四、总结

`write-a-skill` 这个 skill 本身就是一个极佳的示范——它用简洁的篇幅（不到 100 行核心内容）完整地阐述了 skill 的创作方法论，没有一句废话。

**最值得借鉴的三个核心点：**

1. **Description 即入口**：把 skill 的「触发逻辑」作为第一等公民来设计，让 agent 能够准确选择
2. **流程内嵌 QA**：把审查清单融入工作流，而不是作为附录，确保每个交付物都经过验证
3. **渐进式披露架构**：核心文件简洁，扩展内容按需加载，平衡了可用性和完备性

这三个原则不仅适用于写 skill，也适用于写任何工具性文档——先让人会用，再让人用好。
