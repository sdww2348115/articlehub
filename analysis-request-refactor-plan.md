# Skill 深度分析：request-refactor-plan

> 来源：[mattpocock/skills](https://github.com/mattpocock/skills)

## 一、Skill 的作用和核心功能

### 1.1 定位与目标

`request-refactor-plan` 是一个**结构化重构规划技能**。它的核心目标是：**通过系统性访谈和代码探索，将一个模糊的重构需求，转化为一组可供执行的小步骤提交计划，并最终归档为 GitHub Issue**。

简单来说，这是一个"AI 辅助的重构 RFC（Request for Comments）生成器"。

### 1.2 核心功能拆解

整个 Skill 遵循一个清晰的工作流：

```
用户发起重构请求
    ↓
① 深度访谈（收集问题背景）
    ↓
② 代码探索（验证假设、理解现状）
    ↓
③ 方案对比（考虑其他选项）
    ↓
④ 实施方案访谈（深入细节）
    ↓
⑤ 明确边界（确定做什么 / 不做什么）
    ↓
⑥ 测试覆盖率检查
    ↓
⑦ 拆解为微小提交（Martin Fowler 原则）
    ↓
⑧ 生成标准化 GitHub Issue
```

这个流程本质上是一个**结构化决策过程**——从发散的需求收集，到收敛为可执行计划。

### 1.3 解决的问题

| 痛点 | Skill 如何解决 |
|------|----------------|
| 重构范围不清，无限蔓延 | 步骤⑤ 明确 scope 和 out-of-scope |
| 一次性重构风险高 | 步骤⑦ 拆解为"tiny commits"，每步可工作 |
| 测试缺失导致重构失败 | 步骤⑥ 检查测试覆盖率并要求用户明确测试策略 |
| 方案单一、欠考虑 | 步骤③ 主动询问其他方案 |
| 重构计划缺乏文档 | 步骤⑧ 输出标准化的 GitHub Issue |

## 二、写得好在哪里（设计亮点与写作技巧）

### 2.1 结构设计亮点

#### 亮点一：迭代式访谈而非一次性输出

Skill 采用了**多轮访谈结构**（步骤①→④），而不是直接让用户描述问题后立刻生成方案。这是一种"先理解，再方案"的工程思维：

- **步骤①**：问"你想要解决什么问题"，获取高层意图
- **步骤④**：深入访谈"实施方案细节"，获取低层细节
- **步骤③**：补充问"你考虑过其他方案吗"，避免思维盲区

这种设计体现了 **"ask before telling"** 的原则——Agent 在充分收集信息前不贸然下结论。

#### 亮点二：主动降低风险的检查点

步骤⑥专门检查测试覆盖率，如果测试不足，会主动询问用户的测试计划。这是**工程严谨性**的体现——重构而无疑似错误（regression）保障，是高风险行为。

#### 亮点三：Martin Fowler 原则的直接引用

步骤⑦明确引用了 Martin Fowler 的话：

> "make each refactoring step as small as possible, so that you can always see the program working."

这种**引用权威工程实践**的方式，让 Skill 的设计决策有据可查，也提升了说服力。

#### 亮点四：Issue 模板的结构化设计

生成的 GitHub Issue 包含 7 个明确分区：

1. **Problem Statement** — 开发者视角的问题描述
2. **Solution** — 开发者视角的解决方案
3. **Commits** — 微小提交计划（核心产出）
4. **Decision Document** — 决策记录（架构、接口、Schema 等）
5. **Testing Decisions** — 测试策略
6. **Out of Scope** — 明确边界
7. **Further Notes** — 自由补充

这种分区的设计，使得 Issue 本身就是一个**可执行的项目文档**，而非泛泛而谈的需求描述。

### 2.2 写作技巧

#### 技巧一：约束与自由度的平衡

Skill 指令中写道：

> "You may skip steps if you don't consider them necessary."

这在保持结构完整性的同时，**赋予了 Agent 灵活判断的空间**，避免了教条化执行。这是一种"指导而非规定"的设计哲学。

#### 技巧二：Decision Document 中明确禁止代码片段

> "Do NOT include specific file paths or code snippets. They may end up being outdated very quickly."

这个约束非常精准——它预判了一个常见错误：计划文档中出现具体文件路径和代码，会因为代码演进而迅速过期，使文档失去参考价值。**要求只描述意图、接口、架构决策**，让文档更持久。

#### 技巧三：Testing Decisions 的元测试视角

步骤要求描述"什么样的测试是好测试"：

> "only test external behavior, not implementation details"

这体现了对**测试质量**的追求——测试实现细节会导致重构时测试成为阻碍而非保障。这种元测试（test the test）的意识在 Skill 设计中非常难得。

#### 技巧四：Problem Statement 和 Solution 强调"从开发者视角"

Issue 模板中明确要求从开发者视角描述问题和方案，而非系统视角或架构视角。这是**共情设计**——让阅读 Issue 的人（可能是团队其他成员）能快速理解开发者的实际痛点，而非抽象的技术术语。

## 三、对我们构建 Skill 的启示与借鉴

### 3.1 流程设计层面

#### 启示一：多阶段访谈 > 单次问答

我们的一些 Skill 可能过于"急于求答案"，直接让用户描述需求就生成结果。`request-refactor-plan` 提醒我们：**对于复杂任务，分阶段、多轮次的收集-验证-确认循环，能显著提升输出质量**。

**借鉴方式**：在复杂任务 Skill 中，引入"收集→验证→细化"的多阶段流程，而不是一步到位。

#### 启示二：主动管理风险

步骤⑥（测试覆盖率检查）是整个流程中唯一的"风险检查点"。这种在执行前识别风险的意识，值得引入更多 Skill 中。

**借鉴方式**：在涉及代码修改、数据变更、部署等高风险操作的 Skill 中，增加类似的风险检查步骤。

#### 启示三：输出物持久性设计

Decision Document 中禁止文件路径和代码片段，这个约束值得引入我们所有"输出文档类"的 Skill 中。它提醒我们：**文档是给人读的，且人会在未来某个不确定的时间点读**。

### 3.2 模板设计层面

#### 启示四：Issue/PR 模板是 Skill 的核心产出形式

这个 Skill 的最终产出是一个高质量的 GitHub Issue。这给我们一个重要启发：**如果一个 Skill 的产出需要被后续使用或团队共享，以 Issue/PR 模板的形式交付是极好的选择**——结构化、标准化、可操作。

**借鉴方式**：规划类、审查类、分析类的 Skill，可以参考这个 Issue 模板的结构，设计自己的输出模板。

#### 启示五：Out of Scope 分区值得借鉴

在几乎所有规划类文档中，"不做什么"往往比"做什么"更难明确，但价值更高。我们可以**将 Out of Scope 作为所有规划文档的必选分区**。

### 3.3 指令编写层面

#### 启示六：引用权威实践增加可信度

Martin Fowler 的直接引用，让 Skill 的设计理念更具权威性。在我们的 Skill 中，**适当引用公认的最佳实践（如《重构》书籍、某架构模式等），可以增强 Skill 设计的专业性**。

#### 启示七：约束即智慧

Skill 中多处体现了"知道不做什么"的智慧：

- "You may skip steps if you don't consider them necessary." — 约束过度执行
- "Do NOT include specific file paths or code snippets." — 约束文档内容
- "only test external behavior, not implementation details" — 约束测试范围

这些约束看似限制，实则是**把前人踩坑经验编码进 Skill**。

### 3.4 整体架构层面

#### 启示八：Skill 的"三段式"结构

纵观整个 Skill，可以抽象出一个通用架构：

```
[阶段一：理解] — 访谈、探索、验证
    ↓
[阶段二：决策] — 方案对比、边界确定、风险评估
    ↓
[阶段三：产出] — 标准化输出（Issue/PR/报告）
```

这个"理解→决策→产出"的三段式结构，可以作为**我们构建复杂 Skill 时的通用框架**。

## 四、潜在改进建议（批判性思考）

### 4.1 缺少"执行验证"环节

Skill 的流程止步于"创建 Issue"，但没有机制确保 Issue 中的计划被实际执行。是否可以增加一个**后续检查点**，比如询问用户"执行过程中遇到什么问题"？

### 4.2 没有版本演进机制

重构计划在执行过程中可能需要调整，但 Skill 没有设计"计划更新"的流程。这可能超出 Skill 本身的范围，但值得思考。

### 4.3 对多成员协作支持有限

Issue 模板中的 Decision Document 是一个很好的协作工具，但如果能增加"需要团队共识的决策列表"（需要与哪些人达成一致），会更加完整。

## 五、总结

`request-refactor-plan` 是一个**工程思维极其扎实**的 Skill 范例。它的核心价值不在于使用了多么高级的技术，而在于：

1. **把一个模糊的重构需求，通过结构化流程，转化为可度量的执行计划**
2. **将 Martin Fowler 等权威工程实践，编码进 Skill 的决策逻辑中**
3. **以标准化 GitHub Issue 作为最终交付物，实现"规划即文档"**

对于我们构建 Skill 而言，这个 Skill 最重要的借鉴意义在于：**好的 Skill 不是告诉用户"怎么做"，而是引导用户思考清楚"做什么、为什么做、以及不做什么"**。
