# Skill 深度分析：design-an-interface

> 来源：mattpocock/skills  
> 分析日期：2026-04-28

---

## 一、Skill 的作用和核心功能

### 1.1 定位与目的

`design-an-interface` 是一个**接口设计探索型 Skill**，其核心使命是帮助用户在实现之前**系统性地探索同一模块的多种不同设计方案**。它将 John Ousterhout 在《A Philosophy of Software Design》中提出的"Design It Twice"（设计两次）方法论，转化为一套可操作的 AI Agent 工作流。

这个 Skill 不写代码实现，只关注**接口的"形状"（shape）**——即模块暴露的方法签名、参数结构、返回值类型，以及这些元素如何组合成调用者眼中的"门面"。

### 1.2 核心功能拆解

整个 Skill 的工作流分为五个阶段，逻辑递进非常清晰：

| 阶段 | 名称 | 核心动作 |
|------|------|----------|
| 1 | Gather Requirements | 收集需求，厘清问题域和调用方 |
| 2 | Generate Designs | 并行生成 3+ 个**本质不同**的方案 |
| 3 | Present Designs | 逐一展示每个方案 |
| 4 | Compare Designs | 对比分析各方案利弊 |
| 5 | Synthesize | 综合最优，询问用户偏好 |

**关键设计决策**：阶段 2 使用并行 sub-agent 生成多个方案，而非顺序迭代。这不是出于效率考虑，而是为了**保证方案之间的真正独立性**——如果顺序生成，后一个方案容易受前一个方案"锚定"，最终只是小修小补而非真正不同的思路。

### 1.3 适用场景

Skill 的 description 中明确定义了触发条件：

- 用户想设计一个 API
- 探索接口选项
- 比较不同模块形态
- 明确提到"design it twice"

这些场景的共同特征是：**需求已经相对清晰，但接口形态尚未确定**。这不是需求分析阶段，而是需求确认后、实现前的设计决策阶段。

## 二、写得好的地方（设计亮点、写作技巧）

### 2.1 写作结构：工作流驱动的线性叙事

Skill 采用了清晰的**工作流结构**，每个阶段都有明确的目标、具体步骤和产出物。用户不需要任何背景知识就知道"下一步做什么"。这种"步骤清单"式的写作风格，对于 AI Agent 执行任务而言，远比散文式的描述更容易被准确解析。

**亮点细节**：每个步骤都给出了**具体的问题或指令模板**（如 "Ask: 'What does this module need to do? Who will use it?'"），而不是泛泛说"收集需求"。这使得 AI 在执行时能够直接复用这些模板，减少了"如何表达"的决策负担。

### 2.2 并行 Sub-Agent 的约束设计（最精彩的部分）

阶段 2 的设计是整个 Skill 最见功力的地方。它不仅说"生成多个方案"，而是通过**给每个 Agent 分配不同的正交约束**来保证多样性：

- Agent 1：极简主义——最多 1-3 个方法
- Agent 2：最大化灵活性——支持尽可能多的用例
- Agent 3：最优通用情况——为最常见场景优化
- Agent 4：借鉴特定范式——从某个库/ paradigm 获取灵感

这四个约束恰好覆盖了接口设计中最核心的**权衡轴**（trade-off axes）：简洁性 vs 灵活性 vs 性能 vs 表达力。而且这四个维度是**正交的**，能最大化方案之间的差异度。

### 2.3 输出格式的结构化约束

对于每个设计方案，Skill 强制要求输出四个部分：

1. **Interface signature**（类型/方法签名）
2. **Usage example**（调用示例）
3. **What this design hides internally**（隐藏了什么）
4. **Trade-offs**（权衡点）

这个结构保证了不同 sub-agent 的输出在**同一个语义框架**下，便于后续横向比较。"隐藏了什么"这一维度尤其巧妙——它引导设计者思考**信息隐藏**（information hiding），这是模块化设计的核心原则。

### 2.4 对比阶段的"散文优先"原则

> "Discuss trade-offs in prose, not tables."

这一句话堪称整个 Skill 的**点睛之笔**。表格适合展示已知的、可量化的维度对比，但设计权衡往往是**因果链式的、非线性的**——"因为 A 所以 B，进而导致 C"这种叙事，用表格反而会丢失上下文。强制用散文讨论，迫使 AI 写出**有深度的设计推理**，而非机械的行列对照。

### 2.5 评价标准的理论锚点

Skill 引用了《A Philosophy of Software Design》中的四个评价维度：

- Interface simplicity
- General-purpose
- Implementation efficiency
- Depth

这些不是 Skill 作者自己发明的标准，而是来自软件设计领域的经典文献。这种**引用权威来源**的做法，让 Skill 的评价体系有了坚实的理论基础，也更容易被有软件工程背景的用户认可和信任。

### 2.6 Anti-Patterns 的反向约束

Anti-Patterns 段落用**否定式写作**明确了边界：

- 不要让方案趋同
- 不要跳过对比阶段
- 不要实现（只做接口设计）
- 不要按实现难度评价

这是对"工作流police"的有力补充——用户可能会出于惯性走捷径，Anti-Patterns 通过明确**不该做什么**来保护工作流的完整性。

### 2.7 收尾的 Synthesize 阶段

Synthesize 阶段没有武断地宣布哪个方案最优，而是通过两个问题引导用户参与决策：

- "Which design best fits your primary use case?"
- "Any elements from other designs worth incorporating?"

这体现了 Skill 作者的一个隐性判断：**设计决策最终是用户的责任**，AI 应该提供信息和分析，而不是替用户做价值判断。同时，"综合最优"（取各方案之长）的思路，也比"选出一个赢家"更符合实际工程中的做法。

## 三、为我们构建 Skill 能带来什么提示/借鉴

### 3.1 Skill 的本质是"约束下的创造力放大器"

`design-an-interface` 的核心价值，不在于它告诉 AI"怎么设计"，而在于它**强制要求 AI 同时考虑多种可能性**，然后进行比较。这揭示了一个重要的 Skill 设计原则：

> **Skill 的作用不是替代人类的创造力，而是通过结构化的约束，迫使 AI 充分探索方案空间，避免过早收敛。**

这对我们的启示是：当我们设计一个 Skill 时，首先要问的不是"这个 Skill 要做什么"，而是"人类在做这件事时容易犯什么错误"——然后用 Skill 结构来对抗那些错误。例如，人类容易在第一个想法上过早停步，我们就在 Skill 中强制要求多方案生成。

### 3.2 Sub-Agent 并行化的工作流设计模式

`design-an-interface` 开创性地使用了**并行 sub-agent + 正交约束**的模式来生成多样化方案。这是一个可以复用的设计模式：

```
单一任务 → 分解为 N 个子任务（每个子任务有不同约束） → 并行执行 → 汇总比较
```

我们可以将这个模式迁移到其他 Skill 中，例如：

- **代码审查 Skill**：并行生成"安全视角"、"性能视角"、"可读性视角"三个审查报告
- **技术方案设计**：并行生成"保守方案"、"激进方案"、"渐进方案"
- **测试策略**：并行生成"happy path"、"边界条件"、"异常处理"三套测试用例

### 3.3 评价标准必须有理论依据

Skill 引用了《A Philosophy of Software Design》作为评价体系的基础。这是一个很好的做法——当我们为某个领域设计 Skill 时，如果该领域有公认的经典框架或原则，**应该显式引用**，而不是自己发明一套标准。这样做的好处：

1. 标准经过时间检验，有较好的完备性
2. 用户可以追溯理论来源，增强信任
3. 避免 Skill 作者的个人偏好过度影响评价体系

### 3.4 "不要做什么"和"要做什么"同等重要

Anti-Patterns 的存在提醒我们：**Skill 中的否定式约束同样需要精心设计**。特别是"Don't evaluate based on implementation effort"这一条，精准命中了人类在设计阶段容易犯的错误——因为实现难度低就倾向于选择某个方案，而不是看方案本身的质量。

我们在设计 Skill 时，应该主动思考：
- 目标用户在执行这个任务时，最可能走哪条捷径？
- 哪些"看起来合理"的错误做法需要被明确禁止？

### 3.5 输出格式的结构化是可控性的基础

Skill 对每个方案的输出格式做了强制约定（signature / example / hides / trade-offs）。这种结构化输出带来两个好处：

1. **后续阶段可预期**：Compare 阶段知道从哪里提取信息来做对比
2. **失败模式清晰**：如果某个 sub-agent 没有按格式输出，AI 自己就能发现

对于需要多阶段协作的 Skill，**强制输出结构**是确保各阶段无缝衔接的关键。

### 3.6 保持 Skill 的"洁癖"——边界要清晰

Skill 非常明确地说 "Don't implement - this is purely about interface shape"。这是一个**边界清晰的 Skill**——它只做一件事，而且只做这一件事。在实际工程中，我们常常倾向于让一个 Skill 做更多事情（"反正用户可能需要"），结果导致 Skill 变得臃肿、职责不清、Prompt 膨胀。

`design-an-interface` 的简洁性提醒我们：**一个有清晰边界的简单 Skill，比一个大而全的复杂 Skill 更可靠、更易维护**。如果要增加功能，通常应该创建一个新的 Skill，而不是扩展现有 Skill。

### 3.7 人机协作的最后一公里

Synthesize 阶段把设计决策权交还给用户，这是一个很好的人机协作范式。Skill 不应该假设自己比用户更了解业务上下文，但它应该确保用户在做决策时，拥有**充分的信息和正确的比较框架**。

这种"AI 提供分析，用户做决策"的分工模式，比"AI 直接给答案"更符合 Agent 作为"副驾驶"的定位，也更容易在实际使用中获得用户的信任。

## 总结

`design-an-interface` 是一个**方法论驱动**的 Skill，它将软件设计中的经典原则（Design It Twice、信息隐藏、深度模块）转化为一套可执行的工作流。其最核心的价值在于：

1. **并行 + 正交约束**的方案生成模式，确保方案多样性
2. **结构化输出 + 散文对比**，保证可分析性和深度
3. **理论锚点**（《A Philosophy of Software Design》）赋予评价体系合法性
4. **清晰的边界意识**——只做接口设计，不越界实现

对于我们构建 Skill 而言，这个 Skill 提供的最重要借鉴是：**好的 Skill 不是告诉 AI 怎么做，而是约束 AI 的行为模式，确保它不会偷懒走捷径。** 约束（constraints）是 Skill 质量的核心。
